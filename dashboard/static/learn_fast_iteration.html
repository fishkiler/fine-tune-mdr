Phase 3 — Rapid Validation

SmolLM2 Fast Iteration

Validate data quality changes in 30 minutes instead of 12 hours. Same feedback loop, 24x faster. Ship better data to your production model.

Why Fast Iteration?

Foundation-Sec-8B takes 4-12 hours per training run on the DGX Spark. If you change domain weights and it makes things worse, you just lost half a day. That's an unacceptable feedback loop for data quality work.

SmolLM2-135M trains in ~30 minutes. Same data pipeline, same ChatML format, same evaluation harness. You get the same directional feedback at 24x the speed.

The key insight: small models are excellent proxies for data quality testing. If your data has formatting issues, truncation problems, or domain imbalance, SmolLM2 will reveal those problems just as clearly as an 8B model would — it just does it in a fraction of the time.

Think of it like proofreading. You don't need a Pulitzer-winning author to spot a typo — anyone who can read will do. SmolLM2 can "read" your data well enough to tell you when something is wrong.

SmolLM2-135M Architecture

SmolLM2-135M is a 135 million parameter model with a Llama-style architecture. It uses the same core building blocks as Foundation-Sec-8B at 8B scale:

Same architectural patterns at different scales. Data lessons transfer directly.

RoPE (Rotary Position Embedding) encodes token positions using rotation matrices, allowing the model to understand relative distances between tokens regardless of absolute position.

SwiGLU is the activation function in the feed-forward layers. It combines a gated linear unit with the Swish activation, giving smoother gradient flow than standard ReLU.

Grouped Query Attention (GQA) shares key-value heads across multiple query heads, reducing memory use while preserving attention quality. This is the same technique that makes 8B models fit on consumer hardware.

Because these are the same architectural patterns, what you learn about data formatting and quality on SmolLM2 transfers directly to Foundation-Sec-8B. Bad data breaks both models the same way.

Setup

The fast iteration config section is already in config.yaml. Default values: LoRA r=8, alpha=16, batch_size=32, learning_rate=2e-4, 3 epochs, 30 minute time limit.

Run with default data
python -m scripts.train_fast
Run with custom data file
python -m scripts.train_fast --data data/export/chatml_train.jsonl
Run without dashboard
python -m scripts.train_fast --no-dashboard

The adapter will be saved to outputs/smollm2_fast/adapter. Each run overwrites the previous one — this is intentional. Fast iteration is disposable; only your full Foundation-Sec-8B runs produce keepers.

The Iteration Loop

This is the core workflow. Each cycle takes about 35 minutes (30 min training + 5 min eval). You can run 10+ cycles in a single day.

1 Change domain weights or add new data
↓
2 python -m scripts.train_fast  (30 min)
↓
3 python eval.py --model outputs/smollm2_fast/adapter
↓
✓ Improved → promote to full run
× No gain → revert, try again
↓
4 Repeat until eval plateaus
↓
5 Run full Foundation-Sec-8B train (4-12h)
Fast iteration de-risks your full training runs. Never spend 12 hours on untested data changes.

The golden rule: never run a 12-hour Foundation-Sec-8B training on data changes you haven't validated with a 30-minute SmolLM2 run first. Treat SmolLM2 as your staging environment and Foundation-Sec-8B as production.

The Key Insight

"If SmolLM2 can't learn from your data, Foundation-Sec-8B won't either.
But if SmolLM2 learns well, Foundation-Sec-8B will learn MUCH better."

Think of it this way: SmolLM2 is a litmus test for data quality. It won't tell you the final accuracy you'll get at 8B scale, but it will reliably flag problems. A 135M model struggling with your data is a red flag. A 135M model showing clear improvement after a data change is a green light.

Don't compare: SmolLM2 loss 2.1 vs Foundation-Sec-8B loss 0.8. These numbers are meaningless across model scales.

Do compare: SmolLM2 run A loss 2.1 vs SmolLM2 run B loss 1.9. That 10% improvement from your data change will likely translate directionally to the 8B model.

Reading Eval Results
TTP Accuracy

TTP stands for Tactic, Technique, Procedure — the MITRE ATT&CK framework's hierarchy for categorizing adversary behavior. TTP accuracy measures how well your model identifies and classifies these attack patterns.

When you run eval, you'll see accuracy broken down by category: technique identification, tactic mapping, procedure description, detection rule generation, and more. Focus on the categories that matter most to your MDR use case.

Comparing Runs

The real power of fast iteration is A/B testing data changes. Here's a concrete example:

Run A: siem_queries weight = 8 (current)
SPL generation accuracy: 34%
Sigma rule matching: 71%
Run B: siem_queries weight = 12 (+50%)
SPL generation accuracy: 41% ↑
Sigma rule matching: 74% ↑
If bumping siem_queries weight improves SPL generation on SmolLM2, it will likely help Foundation-Sec-8B too.

Watch for regressions. If improving one domain tanks another, you have a domain balance problem. SmolLM2 is small enough that these tradeoffs show up clearly — at 8B scale, the model has enough capacity to mask minor imbalances.

Hands-On
Example Workflow

Let's walk through a real scenario: you notice Sigma rule generation is weak in your latest Foundation-Sec-8B eval. Instead of guessing and retraining for 12 hours, you test the fix in 30 minutes.

Step 1: Baseline fast run

Run a fast iteration with current settings to establish a reference point.

python -m scripts.train_fast
Step 2: Evaluate baseline
python eval.py --model outputs/smollm2_fast/adapter

Note the Sigma rule accuracy. Let's say it's 58%.

Step 3: Bump sigma_rules weight

Edit config.yaml and change sigma_rules weight from 10 to 15:

# In config.yaml, under domain_weights: sigma_rules: 15 # was 10
Step 4: Re-export and retrain
python -m scripts.export_training_data python -m scripts.train_fast
Step 5: Evaluate again
python eval.py --model outputs/smollm2_fast/adapter

Sigma rule accuracy is now 70% — a 12% improvement. Check that other domains didn't regress significantly.

Step 6: Promote to production

The weight change works. Keep sigma_rules at 15 in config.yaml and run the full Foundation-Sec-8B training with confidence:

bash train.sh

You just saved yourself from a potential 12-hour wasted run — or confirmed that your change is worth the investment.
