Phase 6 — Inference

TurboQuant

KV cache compression and inference optimization. Training makes the model smart — this phase makes it fast.

1 Why Inference Optimization?

Training makes the model smart. Inference makes it fast. Once you've trained Foundation-Sec-8B and your game adapters from Phases 3-5, you need to serve them efficiently to real users and agents.

The KV cache is the memory bottleneck for long-context inference. Every token your model has ever seen in the current conversation must be stored in memory so the model can attend to it. For security workloads — analyzing Splunk logs, reviewing code, processing TARS memory — context windows get long fast.

Your DGX Spark has 128GB unified memory. That needs to fit: model weights + KV cache + activations + OS overhead. The KV cache is the variable that grows with every token generated, and it's the one we can compress.

2 The KV Cache Explained
What it stores

During autoregressive generation, the model produces one token at a time. For each new token, it needs to compute attention against every previous token. Without caching, this means re-running the full sequence through every layer on every step — quadratic cost.

The KV cache stores the Key and Value projection vectors from every previous token in every layer. On the next step, the model only computes Q/K/V for the new token, then looks up the cached K and V vectors for all prior tokens. This makes generation linear instead of quadratic.

How big is it?

KV cache size = tokens x layers x heads x head_dim x 2 x bytes_per_param

The "x 2" is for both K and V. Here's what that looks like for Foundation-Sec-8B (32 layers, 8 KV heads, head_dim 128) in BF16:

Sequence Length Tokens KV Cache (BF16) Impact
Short chat 512 64 MB Negligible
Code review 4,096 512 MB Noticeable
Full context 8,192 1 GB Significant
Qwen3.5 max 262,144 32 GB Enormous

At 8,192 tokens on Foundation-Sec-8B, the KV cache alone uses ~1 GB per session. Run 10 concurrent sessions and that's 10 GB just for the cache. On a long-context model like Qwen3.5 at max context, it's 32 GB for a single session — that's a quarter of your total memory.

3 Quantization Fundamentals
Precision levels

Every number in a neural network is stored at some precision. Lower precision = less memory = faster math, but potentially less accuracy.

Format Bits Bytes/Value Compression Quality Impact
FP32 32 4 1x (baseline) Full precision
FP16 / BF16 16 2 2x Minimal loss
INT8 8 1 4x Slight loss
INT4 4 0.5 8x Noticeable for weights
Lower precision = smaller + faster, but may lose information. The sweet spot depends on what you're quantizing.
Weight quantization vs KV cache quantization

Weight quantization is well-established. You've seen it: GGUF Q4_K_M, GPTQ, AWQ. These compress the model's learned parameters (weights) to 4-8 bits. The weights are static — they don't change during inference — so you can calibrate carefully.

KV cache quantization is newer and different. The cache is dynamic — it grows with every token. You can't calibrate it ahead of time because the values depend on the input. The challenge is compressing on-the-fly without losing the information the model needs for accurate attention.

Think of it this way: weight quantization is like compressing a JPEG once. KV cache quantization is like compressing a live video stream — you need it to be fast, adaptive, and reversible enough that the model can still "see" clearly.

4 TurboQuant Concept
The breakthrough

TurboQuant compresses the KV cache to 3-4 bits with zero accuracy loss and zero retraining. It's a drop-in optimization that works on any transformer model at inference time.

5 Your Current Inference Stack
What you have now

vLLM serving two models on the DGX Spark:

# Foundation-Sec-8B on port 8080 export CUDA_VISIBLE_DEVICES=0 python -m vllm.entrypoints.openai.api_server \ --model /path/to/Foundation-Sec-8B-merged \ --port 8080 \ --gpu-memory-utilization 0.85 \ --dtype bfloat16 \ --max-model-len 8192
# Qwen3.5-9B on port 8081 export CUDA_VISIBLE_DEVICES=0 python -m vllm.entrypoints.openai.api_server \ --model /path/to/Qwen3.5-9B \ --port 8081 \ --gpu-memory-utilization 0.85 \ --dtype bfloat16 \ --max-model-len 32768
Why gpu-memory-utilization=0.85

This is critical on the DGX Spark. The NVRM driver deadlocks on OOM — if a CUDA allocation fails, the NVIDIA driver holds a GPU semaphore and blocks nvidia-modeset (your display), freezing the entire system. The Linux OOM killer cannot help because the deadlock is in a kernel module.

Setting gpu-memory-utilization=0.85 tells vLLM to cap its memory pool at 85% of GPU memory. This gives you a clean Python OOM error instead of a system freeze if you exceed the budget.

6 When It Arrives
Current status

TurboQuant is presenting at ICLR 2026. The paper is public, but production-ready implementations are not yet merged into serving frameworks.

What to watch for

Watch for a vLLM pull request adding TurboQuant support. When it merges, the change will likely be a single flag:

# Expected future usage (not yet available) python -m vllm.entrypoints.openai.api_server \ --model /path/to/Foundation-Sec-8B-merged \ --port 8080 \ --gpu-memory-utilization 0.85 \ --dtype bfloat16 \ --max-model-len 8192 \ --kv-cache-dtype turboquant-4bit

This is a drop-in upgrade. No retraining needed. No model changes. No adapter modifications. You just add the flag and your KV cache shrinks by 4-6x.

Why this matters for security workloads

Security analysis is inherently long-context. Analyzing a 500-line Splunk query result, reviewing a full code file for vulnerabilities, maintaining TARS agent memory across a campaign — these all push context windows to their limits. TurboQuant directly addresses the memory constraint that limits how much context you can process per session.
