Phase 5 — Reinforcement Learning

Defender RL

Teach an agent to play Atari Defender using PPO. Learn how trial-and-error reward signals train policies — the same triage instinct a SOC analyst needs.

1 Why RL? Why Defender?

Reinforcement learning teaches agents through trial and error + rewards. There is no labeled dataset. The agent plays, observes outcomes, and adjusts its strategy to maximize cumulative reward over time.

Defender is the perfect training ground. The game demands spatial awareness and priority triage — you must protect human astronauts from waves of aliens, deciding which threats to intercept and which to ignore. You have limited lives and limited time. Sound familiar?

This is exactly what a SOC analyst does: incoming alerts flood in, critical assets need protection, and the analyst must decide which alerts to escalate first. The RL agent learns this triage instinct from scratch, through pure experience.

2 RL Fundamentals
The RL Loop
Agent
(Neural Network)
Action →
Environment
(Defender Game)
Reward + State →
Agent
(Updates Policy)
The agent acts, the environment responds with a reward and next state, and the agent updates its strategy. Repeat millions of times.
Key Concepts

Agent — the neural network that decides what to do. In our case, a CNN that looks at game frames.

Environment — the Defender game. It accepts actions and returns the next frame + a reward signal.

State — the current game frame (what the agent sees). We stack 4 frames so the agent can perceive motion.

Action — joystick direction + fire button. The agent picks one action per step from the discrete action space.

Reward — the game score delta. Destroy an alien = positive reward. Lose an astronaut = negative signal.

Policy — the strategy mapping states to actions. This is what the agent learns. A good policy picks high-reward actions.

Key Insight

Unlike supervised fine-tuning, there are no "correct answers" in RL. Nobody tells the agent "in this frame, press fire." The agent discovers strategy through exploration — trying random actions, observing what gets rewarded, and gradually doing more of what works.

3 PPO Explained
How PPO Works

Proximal Policy Optimization in plain English: the agent plays the game and collects a batch of experiences (states, actions, rewards). Then it updates its policy to do MORE of what got rewards and LESS of what didn't.

The "proximal" part is the safety mechanism: PPO clips updates to prevent the policy from changing too drastically in a single step. Without this, one lucky batch could cause the agent to overcommit to a narrow strategy and forget everything else — catastrophic forgetting.

Without Clipping
PPO Clipping
PPO keeps updates within a safe zone. Small, steady improvements instead of wild swings.
The Driving Analogy

Learning to drive: you make small steering adjustments based on road feedback. PPO is this approach — gentle corrections that keep you in the lane. Without clipping, it would be like yanking the wheel hard every time you drift slightly. You might correct the drift, but you'd also overcorrect into the other lane.

4 Key PPO Metrics

These are the signals to watch during training. Each tells you something different about how the agent is learning.

Policy Loss
should decrease, then stabilize

How much the policy is changing. Early on it drops as the agent finds better strategies. Once converged, it flattens — the agent has settled on its approach.

Value Loss
should decrease

How accurately the value network predicts future rewards. Lower = the agent better understands which states are valuable. A high value loss means the agent is surprised by outcomes.

Entropy
should decrease over time

How random the agent's decisions are. High entropy = exploring (trying many actions). Low entropy = exploiting (confidently choosing specific actions). It should decrease as the agent becomes more decisive, but if it drops to near-zero too fast, the agent may have converged prematurely.

KL Divergence
should stay small

How much the new policy differs from the old one after an update. PPO aims to keep this small. A spike means the update was too aggressive — the clipping mechanism should prevent this, but watch for sustained high values.

Clip Fraction
0.1 – 0.2 is healthy

How often updates get clipped (prevented from being too large). 0.1–0.2 is healthy — it means the clipping is active but not dominating. Near 0 means updates are too small (learning slowly). Near 1 means almost every update is being clipped (learning rate may be too high).

5 Environment Setup
Atari Wrappers

The raw Atari environment needs several wrappers to make RL training efficient. Each one solves a specific problem:

VecTransposeImage — channels-first for CNN
↑
VecFrameStack(4) — stack 4 frames for motion
↑
AtariWrapper — frame skip + terminal on life loss
↑
Raw Atari Env — DefenderNoFrameskip-v4
Wrappers stack from bottom to top. Each transforms the observation or reward before passing it up.

AtariWrapper — skips every 4th frame (the agent doesn't need 60fps decisions), and treats losing a life as terminal. This makes the agent value staying alive.

VecFrameStack(4) — stacks 4 consecutive frames together. A single frame is a photo — no motion info. Four frames show direction and speed of aliens and the ship.

VecTransposeImage — rearranges image dimensions from HWC to CHW format. PyTorch CNNs expect channels first.

Parallel Environments

We run 8 parallel environments simultaneously. Each one plays a different game, collecting diverse experience. More environments = more varied training data per update step, which stabilizes learning.

Config Values
n_envs: 8 # parallel game instances n_steps: 128 # steps per env before update batch_size: 256 # minibatch size for PPO updates # total experience per update = 8 * 128 = 1024 steps
Hands On
6 Training Walkthrough
Start Training
python games/defender.py train # 1M steps (default) python games/defender.py train --steps 2000000 # 2M steps
What to Expect

Early episodes (0–200K steps) — random play. The agent moves erratically, fires without aiming. Reward ~100–500. This is pure exploration — the agent is building its first map of what actions lead to rewards.

Mid-training (200K–600K steps) — basic strategy emerges. The agent learns to move toward aliens and fire. Reward ~2,000–5,000. You'll see the reward curve start climbing steadily.

Converged (600K–1M steps) — skilled play. The agent prioritizes threats, protects astronauts, uses the full screen. Reward ~5,000–10,000. The reward curve flattens — the agent has found its strategy.

Training Time

~18 minutes for 1M steps on DGX Spark. Fast enough to iterate on hyperparameters. You can watch progress live on the /rl dashboard.

7 Reading RL Dashboard Metrics

The /rl dashboard shows live training metrics. Here is what to watch:

Mean Reward (Primary Metric)

100-episode rolling average reward. This is the single most important number. It should trend upward over training. If it plateaus, the agent has converged. If it drops, something went wrong (learning rate too high, or the environment changed).

Episode Length

How long the agent survives per episode. Shorter episodes early on = dying fast (normal). Longer episodes = surviving more waves. If episode length suddenly drops mid-training, the agent may have learned a degenerate strategy.

FPS (Throughput)

Frames processed per second. On DGX Spark: ~937 FPS. This should stay constant. A drop means something is bottlenecking (CPU env stepping, or GPU memory pressure).

Convergence Signals

Training is converged when you see all of these simultaneously:

• Reward curve has plateaued (flat for 100K+ steps)

• Entropy is low (agent is decisive, not exploring)

• Policy loss is stable (no more big swings)

• Clip fraction around 0.1–0.2 (updates are controlled)

8 Play Mode
Watch the Trained Agent
python games/defender.py play
What to Look For

Threat prioritization — does the agent go after the nearest aliens first, or does it chase distant ones? A good agent prioritizes nearby threats that endanger astronauts.

Astronaut protection — does the agent position itself between aliens and the astronauts on the ground? This shows the agent learned that losing astronauts is costly.

Retreat behavior — when overwhelmed by a swarm, does the agent pull back and regroup? Or does it charge in and die? Smart retreat is a sign of long-horizon planning.

Common Failure Modes

• Always moving one direction — the agent found a local optimum (e.g., "flying right gets some reward") and never explored alternatives. Needs more entropy/exploration.

• Ignoring one side of screen — the agent only learned to deal with threats in its immediate view. The CNN may not be processing the full frame effectively.

• Never using hyperspace — hyperspace (emergency teleport) is risky but sometimes necessary. If the agent never uses it, it may not have encountered enough near-death scenarios during training.

9 Connection to MDR

The Defender RL agent and a SOC analyst face the same fundamental problem: prioritize threats under resource constraints. The analogy runs deep:

Defender (Game) MDR (Security)
Alien waves Incoming security alerts
Astronauts to protect Critical assets (servers, data, endpoints)
Limited lives & ammo Limited analyst time & attention
Priority triage Which alert to investigate first
Spatial awareness Understanding attack scope & blast radius
Threat escalation Escalating critical incidents to senior analysts
Hyperspace retreat Isolating a compromised system
Score maximization Minimizing mean time to respond (MTTR)
Multi-wave survival Sustained 24/7 alert monitoring

The RL-trained Defender agent learns to triage — the same skill SOC analysts need. It discovers that protecting high-value targets (astronauts) matters more than chasing every alien. It learns when to fight and when to retreat. These are not programmed rules — they emerge from reward optimization, just like analyst intuition emerges from experience.
