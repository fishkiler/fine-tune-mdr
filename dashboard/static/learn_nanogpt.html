Phase 0 — Foundation

nanoGPT Shakespeare

Build hyperparameter intuition on a model small enough that every change shows visible effects in 3 minutes. One afternoon, seven experiments.

Why Start Here?

Foundation-Sec-8B takes 4-12 hours per training run. If you change a hyperparameter and it makes things worse, you just lost half a day. That's a terrible feedback loop for learning.

nanoGPT on Shakespeare trains in ~3 minutes. You can change learning rate, depth, dropout — and see the effect immediately. Run 7 experiments in an afternoon. By the end, you'll be able to predict what a change will do before running it.

That intuition transfers directly. The hyperparameters are the same dials — just at different scales.

Setup

Clone nanoGPT and prepare the Shakespeare character-level dataset:

git clone https://github.com/karpathy/nanoGPT cd nanoGPT python data/shakespeare_char/prepare.py python train.py config/train_shakespeare_char.py

The last command runs the baseline training. Watch the loss go down. When it finishes (~3 min on GPU), note the final validation loss. That's your reference point for every experiment below.

Experiments
1 Baseline — Run the Defaults
What to change

Nothing. Run the default config as-is.

python train.py config/train_shakespeare_char.py
What to expect

Validation loss around ~1.47. Smooth convergence curve — loss drops quickly at first, then gradually levels off.

Why it matters

This is your anchor. Every experiment below changes one variable from this baseline. If you can't remember the baseline number, you can't evaluate whether a change helped or hurt.

Write down the exact final val loss. It may vary slightly between runs due to random initialization, but it should be close to 1.47 every time.

2 Learning Rate Too High — Loss Explodes
What to change
python train.py config/train_shakespeare_char.py --learning_rate=1e-2
What to expect

Loss explodes. It may spike to NaN or oscillate wildly without converging. The model learns nothing useful.

Why it happens

Learning rate controls the step size of gradient descent. Imagine walking down a valley to find the lowest point. A good step size lets you walk smoothly downhill. Too large? You overshoot the valley floor and end up bouncing between the walls, climbing higher with each step.

At 1e-2, the weight updates are so large that the model jumps over every minimum. The loss goes up instead of down — that's divergence.

3 Learning Rate Too Low — Loss Barely Moves
What to change
python train.py config/train_shakespeare_char.py --learning_rate=1e-5
What to expect

Loss barely decreases. After 5000 iterations, you'll still be far above the 1.47 baseline. The model is learning, but agonizingly slowly.

Why it happens

The steps are too small. The model nudges its weights by tiny amounts each iteration. It would eventually converge, but it might take 100x longer. In practice, you'd run out of patience (or compute budget) long before it gets there.

Key insight: Learning rate is the most impactful hyperparameter. Too high = chaos. Too low = wasted compute. The sweet spot for most LLMs is between 1e-4 and 1e-3, with warmup to ramp into it safely.

4 No Dropout — Overfitting Revealed
What to change
python train.py config/train_shakespeare_char.py --dropout=0.0
What to expect

Training loss drops faster and lower than baseline. But validation loss stops improving earlier and may start rising. The gap between train and val loss widens — that's overfitting.

Why it happens

Dropout randomly deactivates neurons during training (typically 5-20% per step). This forces the network to be redundant — no single neuron can memorize a pattern because it might be switched off next step.

Without dropout, the model is free to memorize the training data exactly. It gets great at reproducing Shakespeare it's seen, but fails on new text. The train/val gap is the hallmark of overfitting.

5 Half Depth — Underfitting
What to change
python train.py config/train_shakespeare_char.py --n_layer=3
What to expect

Higher final loss than baseline (both train and val). Converges faster (fewer parameters to update) but hits a ceiling. The model simply doesn't have enough capacity to learn the full complexity of Shakespeare.

Why it happens

Each transformer layer is a chance for the model to build a more abstract representation. Layer 1 might learn character patterns. Layer 2 learns word fragments. Layer 3 learns common phrases. Layers 4-6 learn grammar and style.

With only 3 layers, you're cutting the model's ability to represent higher-order patterns. It's like trying to describe a painting with only primary colors — you can get the broad strokes, but the nuance is lost.

6 Double Attention Heads — Marginal Gains
What to change
python train.py config/train_shakespeare_char.py --n_head=8
What to expect

Very slight improvement over baseline, or no change at all. The model doesn't suddenly get twice as smart.

Why it happens

Attention heads are like parallel perspectives. Each head can attend to different aspects of the input — one might focus on nearby characters, another on punctuation patterns, another on word boundaries.

With 4 heads, you already have decent coverage. Doubling to 8 gives more perspectives, but the total attention dimension stays the same (it's split among more heads). Each head becomes narrower. The net effect is often a wash at this model scale.

Key insight: Attention heads matter more at scale. In GPT-3 (96 heads) vs a 4-head model, the difference is dramatic. At nanoGPT scale, depth tends to matter more than head count.

7 Tiny Batch Size — Noisy Gradients
What to change
python train.py config/train_shakespeare_char.py --batch_size=16
What to expect

The loss curve is much noisier — it jitters up and down instead of smoothly declining. Final loss may be similar to baseline, but the path there is bumpy.

Why it happens

Each training step computes gradients on a batch of examples. A large batch gives a stable estimate of the true gradient direction. A small batch is like polling 16 people instead of 64 — the answer is noisier.

Small batches aren't always bad. The noise can help escape local minima (shallow traps in the loss landscape). But too small and the training becomes unstable. There's also a throughput tradeoff — GPUs are optimized for large parallel operations, so tiny batches may actually train slower in wall-clock time.

Key insight for DGX Spark: Your GB10 has 128GB unified memory. You can afford large batches (32+). The optimal batch size from your benchmarks was 16 with seq_length 1024 — that's the sweet spot where throughput peaks without OOM.
