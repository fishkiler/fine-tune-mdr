Phase 2 — Production Training

Foundation-Sec-8B SFT

Supervised fine-tuning takes a pre-trained language model and teaches it cybersecurity. LoRA adapters, domain-weighted data, and a full training pipeline on the DGX Spark.

1 Why Supervised Fine-Tuning?

Foundation-Sec-8B already knows language — it can complete sentences, follow grammar, and reason about text. But it doesn't know how to write Sigma rules, analyze attack logs, or generate SIEM queries. SFT bridges that gap.

Supervised fine-tuning feeds the model thousands of (prompt, response) pairs and adjusts its weights so it produces responses like the ones in your dataset. It's the same mechanism as pre-training, but on curated domain-specific data.

Teacher Forcing

During training, the model sees the correct answer at every step — this is called teacher forcing. Instead of generating freely (where errors compound), we feed the ground-truth tokens as input and ask "given this correct prefix, predict the next token." This makes training stable and fast.

Loss on Assistant Tokens Only

In a ChatML conversation, we only compute loss on the assistant's response tokens. The system prompt and user message are context — we don't penalize the model for "predicting" those. This focuses all gradient signal on the behavior we want to teach.

<|system|> → <|user|> Analyze this log... → <|assistant|> The log shows...
no loss → no loss → loss computed here
Loss is only computed on assistant tokens. System and user tokens are masked — they provide context but don't contribute to gradients.

Key insight: SFT doesn't teach the model new knowledge from scratch. It teaches the model to surface and organize what it already knows in a cybersecurity-specific way. The pre-training gave it the vocabulary; SFT teaches it the job.

3 LoRA Explained

Low-Rank Adaptation (LoRA) is a technique that makes fine-tuning feasible on consumer hardware. Instead of updating all 8 billion parameters, we freeze the original weights and inject small trainable matrices alongside them.

The Core Idea

A pre-trained weight matrix W has dimensions d×d (e.g., 4096×4096 = ~16M parameters). LoRA decomposes the update into two small matrices: B (d×r) and A (r×d), where r is the rank — typically 8-64.

During fine-tuning, the effective weight becomes W' = W₀ + B×A. Only B and A are trained. W₀ stays frozen.

W₀ has 16.7M params (frozen). B×A has only 131K params (trainable). That's 0.8% of the original — roughly 100x fewer parameters.
Key Hyperparameters

r (rank) = 16 — Controls the expressiveness of the adaptation. Higher rank = more capacity but more parameters. 16 is a good balance for domain adaptation.

alpha = 32 — Scaling factor applied as alpha/r. With alpha=32 and r=16, the LoRA update is scaled by 2x. Think of it as the "learning intensity" for the adapter.

target_modules = q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj — We apply LoRA to all attention projections (Q/K/V/O) and the MLP layers (gate/up/down). This covers the full transformer block, giving LoRA the ability to modify both what the model attends to and how it transforms information.

Why Not Full Fine-Tuning?

Full fine-tuning of 8B parameters requires ~64GB just for optimizer states (Adam stores 2 copies). LoRA trains ~42M parameters instead — optimizer states fit in ~500MB. On the DGX Spark with 128GB unified memory, this is the difference between a comfortable fit and running out of memory.

Why So Many Steps?

Garbage in, garbage out applies double for fine-tuning. A single malformed Sigma rule in the training set teaches the model to generate malformed Sigma rules. The pipeline catches format errors (validate), quality issues (review), and ensures the right domain balance (export with weights).

The --new-only flag on export means you can incrementally add data without re-exporting everything. Each record gets an exported_at timestamp so you know exactly what the model was trained on.

5 Domain Weight Tuning

The training dataset spans multiple cybersecurity domains with wildly different sizes. Without balancing, the model would see 618K CVE records for every 348 code reviews — it would become an expert at CVE descriptions and terrible at everything else.

Domain Weights
Domain Weight Records Rationale
cve 0.08 ~618,000 Downsample heavily — most are repetitive boilerplate
mitre_attack 10.0 ~1,317 Oversample 10x — high signal, ATT&CK framework critical
secure_code_review 30.0 ~348 Oversample 30x — extremely high signal, very few examples
siem_queries 8.0 ~700 SPL/KQL generation critical for Splunk/Sentinel ops
sigma_rules 10.0 ~700 YAML detection rules must be syntactically perfect
log_analysis 5.0 ~1,600 Direct MDR relevance — analyzing real attack logs
How Weights Work

During export, each domain's records are sampled proportionally to their weight. A weight of 30.0 means a record from that domain is 30x more likely to appear in the final training set compared to weight 1.0. A weight of 0.08 means aggressive downsampling — only ~8% of CVE records make it through.

The result: instead of 99% CVEs, the training set has a balanced mix where the model sees code reviews and Sigma rules almost as often as CVE descriptions, despite having 1,700x fewer source records.

Key insight: Domain weighting is one of the highest-leverage decisions in the entire pipeline. Getting these ratios wrong produces a model that's great at one thing and useless at five others.

6 Training Run Walkthrough
Launch Commands
# Default: resumes from latest checkpoint python train_native.py # Explicit config file python train_native.py --config config.yaml # Fresh start: ignore all checkpoints python train_native.py --fresh
What Happens During Training

1. Tokenization — ChatML conversations are tokenized using the model's tokenizer. Each message gets wrapped with special tokens (<|system|>, <|user|>, <|assistant|>). Labels are masked for non-assistant tokens.

2. Packing — Multiple short examples are concatenated into a single sequence up to max_length (1024 tokens). This eliminates ~38% padding waste and is the biggest single optimization for throughput on the GB10.

3. Gradient Updates — Each batch computes the cross-entropy loss on assistant tokens, backpropagates through the LoRA matrices (B and A), and applies the Adam optimizer. The base model weights stay frozen throughout.

4. Checkpoint Saving — Every N steps, the trainer saves LoRA adapter weights (not the full model). Checkpoints are ~170MB each. HF Trainer evaluates before saving — if you kill during eval, that checkpoint is lost.

Key Config Values
# From config.yaml — the important dials per_device_train_batch_size: 16 max_length: 1024 # sequence length after packing learning_rate: 2e-4 # with cosine schedule warmup_ratio: 0.03 # 3% of steps warmup num_train_epochs: 3 gradient_checkpointing: true bf16: true # BF16 faster per-token on GB10

Why BF16? On the GB10 (sm_121 Blackwell), BF16 is faster per-token than 4-bit. bitsandbytes dequantization adds overhead that costs more than the memory savings on unified memory hardware. Use native BF16 and skip quantized training entirely.

7 Dashboard Monitoring

The training dashboard (port 7860) gives you real-time visibility into the training run. Here's what to watch and what the signals mean.

Loss Curve
Good: Smooth decrease, flattening over time
Bad: Sudden spikes, plateaus, NaN
Learning Rate
Good: Warmup ramp → cosine decay to ~0
Bad: Should never be flat (check scheduler)
Gradient Norm
Good: Stable range, gradual decrease
Bad: Sudden spikes = instability or bad data
GPU Memory
Good: Stable plateau after first few steps
Bad: Steady climb = memory leak, approaching OOM
Signs of Trouble

Loss plateau early — Learning rate may be too low, or the model has memorized the small dataset. Check if packing is working (if most sequences are padding, effective batch size is tiny).

Gradient explosion — Grad norm spikes to 100+ then NaN. Usually caused by a corrupted training example or learning rate too high. Check the last batch before the spike.

OOM after N steps — Memory leak. Make sure gradient_checkpointing is enabled. If using eval, the eval set might not fit — set eval_strategy: "no" for quick test runs (31K val set takes ~2.5 hours per eval pass).

Train/val gap widening — Overfitting. Reduce epochs, increase dropout, or add more diverse training data. This is especially common when oversampling small domains aggressively.

8 Post-Training
Free Trainer Memory First

Before merging, free the trainer's memory to prevent peak memory from doubling:

del trainer import gc; gc.collect() import torch; torch.cuda.empty_cache()
Merge LoRA Adapters

LoRA adapters are separate files (~170MB). To use the model without the adapter machinery, merge them back into the base weights:

# Merge happens automatically at end of train_native.py # Or manually: from peft import PeftModel model = PeftModel.from_pretrained(base_model, adapter_path) merged = model.merge_and_unload() merged.save_pretrained("output/merged-16bit")
GGUF Quantization

For deployment with Ollama or llama.cpp, convert to GGUF format:

# Convert merged model to GGUF Q4_K_M python convert_hf_to_gguf.py output/merged-16bit \ --outtype q4_k_m \ --outfile output/foundation-sec-mdr-Q4_K_M.gguf
Evaluation
python eval.py

What to check:

• Per-domain accuracy — Does the model handle Sigma rules as well as SIEM queries? Weak domains indicate weight tuning is needed.

• Format compliance — Are generated Sigma rules valid YAML? Do SIEM queries parse in SPL/KQL?

• Regression — General language quality shouldn't degrade. If the model forgets how to form coherent sentences, you've over-trained.
