autoresearch

Now that you have intuition, let the machine run 100 experiments overnight while you sleep. Wake up smarter.

Prerequisites

Before starting autoresearch, make sure you have:

  • Completed Phase 0: nanoGPT Shakespeare
  • Flash Attention compiled for sm_120 (see below)
  • Environment variables configured for CUDA 13/12 compatibility
  • A tmux or screen session (experiments run overnight)
Flash Attention Build (one-time, ~75 min)
export TORCH_CUDA_ARCH_LIST="12.0" export CUDA_HOME=/usr/local/cuda git clone https://github.com/Dao-AILab/flash-attention.git cd flash-attention python setup.py install

The sm_120 build runs on sm_121 (Blackwell) via forward compatibility. This is the single highest-leverage optimization — unlocks full training throughput for all models.

Environment Variables
export TORCH_CUDA_ARCH_LIST="12.0" export LD_LIBRARY_PATH="${CONDA_PREFIX}/lib/python3.11/site-packages/nvidia/cuda_runtime/lib:${LD_LIBRARY_PATH}" export VLLM_MOE_KERNEL_BACKEND=triton export VLLM_DISABLED_KERNELS=cutlass_moe_mm,cutlass_scaled_mm export TRITON_PTXAS_PATH=/usr/local/cuda-13.0/bin/ptxas
Setup

Clone autoresearch and install dependencies using uv (ultra-fast Python package manager):

git clone https://github.com/karpathy/autoresearch cd autoresearch # Install uv if you don't have it curl -LsSf https://astral.sh/uv/install.sh | sh # Install dependencies and prepare data uv sync uv run prepare.py

prepare.py downloads and tokenizes the training corpus. autoresearch then uses an LLM agent to design, run, and evaluate experiments autonomously.

How autoresearch works: An AI agent reads a program.md file describing what to explore. It designs an experiment, modifies training code, runs it, reads the results, and decides what to try next. It repeats this loop ~100 times overnight.

DGX Spark program.md Additions

Add this to autoresearch's program.md so the agent understands your hardware and what to explore:

Hardware Context
## Hardware context - DGX Spark: NVIDIA GB10 Superchip, sm_121, 128GB unified memory - Flash Attention compiled for sm_120 (forward compatible with sm_121) - CUDA 13.0 — do NOT use torch_compile (recompilation spikes on Blackwell) - SDPA available as fallback: attn_implementation="sdpa" - Muon optimizer works well on this hardware - BF16 is faster than FP16 on Blackwell
Exploration Axes
## What to explore - Architecture changes (depth, heads, FFN multiplier) - Optimizer settings (Muon momentum, AdamW beta2) - Banded vs full attention patterns (WINDOW_PATTERN: "L" vs "SSSL") - Sequence length vs batch size tradeoffs

These constraints prevent the agent from wasting time on things that don't work on your hardware (like torch.compile or FP16), and point it toward the most productive search directions.

Understanding the Exploration Axes
Architecture

Depth (number of layers), width (hidden dimension), heads (attention head count), and FFN multiplier (feedforward network size relative to hidden dim). The agent tries different combinations to find what gives the best loss per parameter.

Common finding: wider + shallower often beats narrow + deep at small scale. The agent discovers this empirically.

Optimizers

AdamW is the standard. Muon is newer and often converges faster. The agent compares them with different learning rates, momentum values, and schedules.

Attention Patterns

Full causal attention is expensive. Alternatives like sliding window or banded patterns (SSSL = short-short-short-long) reduce compute while retaining most quality. The agent measures the tradeoff.

Sequence / Batch Tradeoffs

Longer sequences see more context but use more memory (quadratic in attention). Larger batches give smoother gradients but also use more memory. The agent finds the sweet spot for your 128GB unified memory.

Reading Results
What is val_bpb?

val_bpb = validation bits per byte. It measures how many bits the model needs to predict each byte of text. Lower is better — the model is more efficient at encoding the patterns it learned.

val_bpbQualityInterpretation
> 1.2PoorModel barely learned the data distribution
1.0 – 1.2DecentBasic character/word patterns captured
0.9 – 1.0GoodStrong generation, understands grammar
< 0.9ExcellentNear state-of-the-art for this model scale

Relationship to loss: val_bpb ≈ loss × (ln(2) / 8). If you're used to thinking in cross-entropy loss, just know that lower bpb = lower loss = better.

Patterns to Look For

When you read the morning log, look for these signals:

Consistent Improvements

If the agent tried "increase FFN multiplier from 4x to 6x" three times with different configs, and it helped every time, that's a real signal. If it helped once out of three, it's probably noise or context-dependent.

Diminishing Returns

Watch for the agent repeatedly trying small variations (lr=2.3e-4 vs 2.4e-4 vs 2.5e-4) with minimal improvement. That means it's converged in that dimension — the remaining gains are elsewhere.

Surprising Findings

The most valuable experiments are ones that surprise you. "Smaller depth + wider FFN outperforms deeper + narrower" contradicts the common assumption that more layers = better. If autoresearch finds this, pay attention — it may transfer to Foundation-Sec-8B.

The 3-Run Rule

Machine learning experiments have inherent randomness from weight initialization, data shuffling, and dropout. A single experiment is never enough to draw conclusions.

The rule: If an improvement appears in 3 out of 3 runs with different random seeds, it's real. If only 1 out of 3 shows improvement, it's noise. 2 out of 3 is borderline — run 2 more to be sure.

autoresearch handles this automatically for its best findings. When reviewing results, check whether the top improvements were verified across multiple seeds.

Transfer to Foundation-Sec-8B

Not everything autoresearch discovers will transfer to your larger model. Here's what typically does and doesn't:

FindingTransfers?Why
LR schedule shape (warmup + cosine) Yes Schedule dynamics are scale-invariant
Architecture ratios (FFN/hidden) Yes Optimal ratios are consistent across scales
Optimizer choice (Muon vs AdamW) Often If Muon wins small, it often wins large
Attention patterns (sliding window) Sometimes Benefits depend on sequence length needs
Absolute LR value (e.g. 3.2e-4) No Optimal LR scales with model size
Absolute batch size No Depends on dataset size and memory

Key insight: Transfer the directions (higher FFN ratio helps, Muon converges faster, warmup matters), not the specific numbers. The numbers need to be re-tuned for each model scale.
