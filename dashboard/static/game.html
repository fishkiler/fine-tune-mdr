Step ? Training Step
One optimizer step = one weight update. Each step processes effective_batch_size examples (batch_size × gradient_accumulation_steps). Total steps = (dataset_size / effective_batch) × epochs. More steps means more passes over the data.
--
Loss ? Training Loss
Measures how wrong the model's predictions are. Lower is better. For game adapters, this measures how well the model predicts the correct action given a game frame. Loss should decrease rapidly in the first epoch and then plateau.
<0.2 — Converged, model has learned the mapping
0.2–0.5 — Good, still refining predictions
0.5–1.5 — Normal for early training
>1.5 — High, check data or learning rate
--
Learning Rate ? Learning Rate
Controls how much the model's weights change per step. Uses a cosine schedule: warms up from 0 to peak (e.g. 2e-4) over the first ~100 steps, then gradually decays toward zero. Too high causes instability (loss spikes), too low means the model stops learning. The schedule is automatic — no intervention needed.
--
Speed ? Training Speed
Samples processed per second. For VLM (vision-language) training, speed is limited by image loading and processing through the vision encoder, not just GPU compute. Speed does not affect model quality — it only determines how long training takes. Typical for a 9B VLM on DGX Spark: 1–3 samples/sec.

GPU Memory ? GPU Memory Allocated
How much GPU memory the model, optimizer states, and activations are using. On DGX Spark's unified memory (128 GB shared CPU/GPU), this is the CUDA-allocated portion. Memory usage is mostly constant after the first few steps. If it exceeds 85% of available CUDA memory, the NVIDIA driver can deadlock on OOM — the training script caps usage at 85% to prevent this.
<70% — Plenty of headroom
70–85% — Comfortable
85–95% — Tight, risk of OOM
>95% — Critical, may crash
--
GPU Temp ? GPU Temperature
GPU die temperature from the NVIDIA driver. Training keeps the GPU under sustained load, generating heat. GPUs can safely run hotter than CPUs, but sustained high temps reduce hardware lifespan and may trigger thermal throttling (reducing speed).
<65°C — Cool
65–75°C — Normal under load
75–85°C — Warm, still safe
>85°C — Hot, may throttle

CPU Temp ? CPU Temperature
Hottest reading across ACPI thermal zones (includes VRMs and chipset, not just CPU die). During VLM training the CPU handles data loading, image preprocessing, and feeding batches to the GPU. High CPU temps usually mean the dataloader workers are saturating the CPU.
<60°C — Cool
60–70°C — Normal under load
70–80°C — Warm
>80°C — Hot, check cooling

ETA ? Estimated Time Remaining
Time until training completes all scheduled steps, based on a rolling average of recent step durations. May fluctuate early on (warmup, JIT compilation) and stabilize after ~50 steps. Note: this shows time to scheduled completion — if the model has already converged, the remaining time is wasted compute. Check the Training Report for convergence status.
--
Step Progress ? Step Progress
Percentage of total optimizer steps completed. Total steps = (training_examples / effective_batch_size) × epochs. Each step updates the LoRA adapter weights once. The bar fills linearly — but model quality typically improves logarithmically (fast gains early, diminishing returns later). -- / --
Epoch Progress ? Epoch Progress
One epoch = one full pass through all training examples. Multiple epochs let the model see each example several times, reinforcing learning. However, too many epochs leads to overfitting — the model memorizes training data instead of generalizing. For game adapters with repetitive frame data, 2–3 epochs is usually sufficient. -- / --
GPU Memory ? GPU Memory Usage
CUDA-allocated memory as a fraction of total available GPU memory. Includes model weights (frozen base + trainable LoRA), optimizer states (AdamW keeps 2 copies of LoRA params), gradients, and activations. Gradient checkpointing trades compute for memory by recomputing activations during backward pass instead of storing them. -- / -- GB

Excellent/Good means training is progressing well. Needs Attention means something may be wrong. The insight text at the bottom provides specific, actionable recommendations based on current metrics. Convergence ? Convergence Analysis
Tracks whether the model has finished learning. Current Loss is the latest training loss. Min Loss is the lowest loss seen and the step it occurred at. Loss Trend compares the average loss over the last 10 steps vs the 10 before that — "Plateaued" means <2% change, indicating convergence. Gradient Norm measures the magnitude of weight updates — small norms (<1) mean the model has little left to learn. Convergence Point is the first step where loss dropped below 0.2.

Loss Curve ? Loss Curve
The most important chart for monitoring training. The faint line shows raw per-step loss (noisy), and the bold line shows an exponential moving average (EMA, α=0.1) for the true trend. A healthy loss curve drops steeply in the first epoch, then gradually flattens. If the curve is flat from the start, the learning rate may be too low. If it spikes or oscillates wildly, the learning rate may be too high. Once the EMA line is flat for 500+ steps, the model has converged.
Learning Rate Schedule ? Learning Rate Schedule
Shows the cosine annealing schedule. The LR starts at 0, rises to peak during warmup (first ~100 steps), then smoothly decays following a cosine curve toward 0 by the final step. This schedule helps: high LR early for fast learning, low LR late for fine-grained refinement. The shape should be a smooth arch — any discontinuities indicate a training restart or checkpoint resume.
GPU Memory (GB) ? GPU Memory Over Time
Tracks CUDA memory allocation across training. Memory typically jumps at step 0 (model loading), rises slightly during the first few steps (optimizer state initialization, activation caching), then remains nearly flat. A gradual upward drift may indicate a memory leak. Sudden spikes followed by drops are garbage collection events. If the line approaches the memory cap (set at 85% of total), training is at risk of OOM.
Training Control ? Training Control
Stop sends a SIGINT (cooperative shutdown signal) to the training process. The trainer finishes its current step, saves a full checkpoint (model weights, optimizer state, scheduler state, RNG), then exits cleanly. This checkpoint can be resumed later with --resume. A second SIGINT forces an emergency save and immediate exit. The process badge shows: Idle (not started from dashboard), Running, Stopping (SIGINT sent, saving), Complete, or Error. Idle
Sends SIGINT for cooperative shutdown. Model saves checkpoint before exit.
Training Log ? Training Log
Live training activity feed. Shows a summary line every 10 steps with loss, learning rate, speed, GPU memory, temperature, and ETA. When training is started from the dashboard, also streams full stdout/stderr (model loading, checkpoint saves, errors). Lines are color-coded: red for errors/tracebacks, yellow for warnings, green for success messages (saves, completion), blue for info. Auto-scrolls to latest output.
